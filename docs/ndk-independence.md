# NDK independence and the game-platform direction (plan, 2026-07-19)

Goal: eliminate the proprietary Novell NDK from the build, then grow a small, license-clean
runtime suitable for writing/porting **games** to NetWare 3.x (lots of low-level work: direct
hardware access to graphics, sound, keyboard). Not implemented yet — this captures the analysis
so any session can pick it up.

## What the build actually uses from the NDK (measured, 2026-07-19)

The `/usr/nwsdk` dependency surface is tiny:

1. **`prelude.o` — 892 bytes of pure glue** (disassembled and understood in full):
   - `_Prelude(<9 loader args>)` → forwards all 9 args + `NULL` + `&_cstart_` to CLIB's
     `_StartNLM` (11 args, caller-cleaned).
   - `_cstart_()` → `_SetupArgv(main)` — CLIB builds argc/argv and calls `main` itself.
   - `_Stop()` → `_TerminateNLM(&NLMID, 0, 5)`; `NLMID` is a 4-byte BSS slot.
   - nlmconv's default START/EXIT symbols are `_Prelude`/`_Stop` — a replacement keeping those
     names is a drop-in. (Structure verified by disassembly; the meaning of `_TerminateNLM`'s
     `0, 5` arguments and the NULL are inferred — cross-check against Novell's public developer
     docs before reimplementing.)
2. **One header prototype**: only `nwconio.h` is still included, for `ConsolePrintf`.
   `implicit_nlm_defs.h` already demonstrates the replacement pattern (own declarations for
   `inp`/`outp`/`delay`).
3. **`imports/*.imp` are NOT Novell's** — symbol-name lists generated by Hinner's free `nlmimp`
   from a server's CLIB.NLM (says so in the file header). Interface facts; regenerable.
4. Nothing else on the NDK ISO is used by this project.

## Tiers to NDK-free builds

- **Tier 1 (hours): drop the headers.** Own prototype for `ConsolePrintf` next to
  `implicit_nlm_defs.h`; delete the stale `nwadv.h` include in `vga_util.c`.
- **Tier 2 (a day + boot tests): clean-room `prelude.o` replacement** (~20 lines of C/asm per
  the description above, same symbol names). After this the build is 100% free software
  (GPL binutils, gcc, LGPLv2 nlm-kit, nlmimp-generated `.imp` lists, own headers/prelude), and
  the Dockerfile can drop the NDK ISO download — removing the archive.org availability risk and
  LICENSE.md's grayest entry. CLIB.NLM is still called *at runtime*, but that lives on the
  licensed server, not in the toolchain. Ideal first automated test case for the QEMU sidecar.
- **Tier 3 (research): CLIB-free NLMs** — import NetWare kernel exports directly and skip
  `_StartNLM` entirely. Needed only for NLMs that must run without CLIB.NLM loaded, and it is
  where the game platform naturally ends up (below).

## Game platform: portable mini-libc + kernel-export glue

Three layers, cleanly separable:

1. **Freestanding mini-libc (fully portable, zero OS calls)**: `mem*/str*`, `snprintf`, `rand`,
   `qsort`, fixed-point or soft-float math. See "Choosing the mini-libc" below.
2. **NetWare glue (small, the only NetWare-specific code)**:
   - startup: the tier-2/3 prelude;
   - `malloc` backend: kernel `Alloc`/`AllocateResourceTag` exports (resource tags are
     NetWare's allocation bookkeeping — research their exact 3.x signatures);
   - time: kernel tick counter, or program the PIT directly (ring 0 — allowed);
   - yield: NetWare 3.x scheduling is **cooperative**; a game loop that never yields starves the
     server and may trip the CPU-hog watchdog. The loop must periodically call a yield export
     (`ThreadSwitch`/CLIB or the kernel's relinquish-control export — verify names for 3.x).
3. **Direct hardware (the game's domain, already partly proven)**:
   - graphics: VGA mode setting + VRAM writes work (boot-verified 2026-07-19);
   - keyboard: port 0x60/0x64 polling *conflicts with the console's own handler* — either accept
     stolen keys on a dedicated "game console" box, or hook the keyboard IRQ via the kernel's
     interrupt-registration exports (research; drivers do it, so the mechanism exists);
   - sound: Adlib (pure port I/O — easiest), Sound Blaster (port I/O + DMA controller
     programming + IRQ hook); QEMU emulates both, so the sidecar can exercise sound;
   - restore text mode on exit (see [qemu-vm-debugging.md](qemu-vm-debugging.md)) so the server
     stays usable.

## Choosing the mini-libc (assessed 2026-07-19)

An NLM is, from a libc's point of view, a **bare-metal target**: no POSIX, no Linux syscalls,
you supply the backend hooks. That rules some candidates in and some out (general comparison of
the candidates: [Choosing the Right C Library for Embedded Systems — Inferara, Medium](https://medium.com/@inferara/choosing-the-right-c-library-for-embedded-systems-newlib-picolibc-nanolib-and-dietlibc-8366a4701355);
the NLM-specific constraints below are this project's own findings):

- **picolibc — front-runner.** Built (by the newlib-nano author) for exactly this case: tiny,
  BSD/MIT-licensed, actively developed, modern meson build with cross files (a custom
  i386/"none" cross file can carry the house CFLAGS), configurable to **no TLS**
  (`thread-local-storage=false` — required: the NLM format has no `.tdata`/`.tbss`, nlmconv maps
  only code/data/bss) and no-float stdio. Its tiny stdio needs a single putc-style hook — wire
  it to the text buffer / a kernel output export / serial. Provides malloc implementations that
  take an sbrk-style backend (kernel `Alloc` glue), though a **static arena is the better first
  malloc** for games: deterministic, zero OS glue.
- **PDCLib — license-purity fallback** (CC0/public domain, no attribution surface at all), less
  active and less complete; fine for the string/format/qsort tier.
- **newlib — works but heavier**; its reentrancy structure and default sizes buy nothing here.
  newlib-nano is effectively superseded by picolibc for new work.
- **dietlibc — ruled out**: GPLv2 and structurally Linux-syscall-oriented; wrong layer entirely
  (same reason musl is out).

NLM-specific constraints for whichever library (all enforced or catchable by `verify-nlm`):
compile with the house flags (`-m32 -fno-pic -fno-asynchronous-unwind-tables`,
`-fno-stack-protector`, `-fcf-protection=none` per the watch list in
[nlm-toolchain-notes.md](nlm-toolchain-notes.md)); no TLS; nothing may rely on
`.init_array`/`.ctors` running unless the custom prelude walks them (10 extra lines if ever
needed); prefer fixed-point/soft-float math — 3.x-era servers often have no x87, and NetWare 3's
ring-0 FPU context handling across NLM threads is unverified (soft-float pulls in libgcc
routines; adding `libgcc.a` as an `INPUT` works since nlmconv's `ld -Ur` resolves archives).
Link as a static `libc.a` listed in the `.def`.

## Open research items (verify against primary sources before relying on)

- The NetWare 3.x kernel (SERVER.EXE) public-symbol list and how to produce an `.imp` for it:
  documented "operating system public symbols" in Novell's 3.x developer docs, or dump from a
  live server (nlmimp reads NLMs, SERVER.EXE is not one — different extraction needed).
- Exact 3.x semantics/signatures of: `Alloc`, `AllocateResourceTag`, relinquish-control, and the
  interrupt-hook exports; CPU-hog watchdog behavior on 3.x.
- NetWare 3.x has a **built-in kernel debugger** (console keyboard chord, e.g.
  Alt+Shift+Shift+Esc — verify the exact chord for 3.12) — potentially valuable in the QEMU
  debugging loop.

## Suggested sequencing

QEMU sidecar first (fast iteration is the multiplier for everything else) → tier 1+2 (NDK-free,
verified by sidecar boots) → keyboard input + text-mode restore (makes interactive testing sane)
→ mini-libc port → sound. Direct-hardware graphics already works today.
